## MVP Scope

### 必须交付（M1 框架原型）

| 功能 | 说明 |
|---|---|
| 角色创建界面 | 选择性别 + 选择出身（MVP 实现 3 种：寒门学子 / 边境武将 / 落魄商贾） |
| 赛道选择 | 选择初始人生方向（MVP 实现 2 赛道：官场 / 造反）|
| 资源面板 | 每个赛道显示对应 3-4 种核心资源 + 通用"钱粮" |
| 行动系统 | 月令四时辰制：每月4个时段（朔日/旬中/望日/月末），官场8种行动，造反7种行动 |
| 行动结果 | 每个行动 → 即时资源增减 + 时段加成结算 + 一句结果描述文字 |
| 大型行动 | 每种赛道1个占2时段的大型行动（官场"大案钦查"、造反"挥兵攻城"） |
| 被动事件 | 每回合 0-1 个随机事件，玩家必须选择应对方式（2-3个选项） |
| 连续策略 | 实现2个Buff（朝堂重臣 + 粮满仓） |
| 时间叙事 | 回合显示为"建元X年·X季"，回合间有过渡语 |
| 回合结算 | 资源自然增减（基于已选赛道）→ 胜负判定 |
| 简单结局 | 资源归零 → 覆灭结局；20 回合后 → 简单终局结算文字 |

### 可选交付（时间允许时加入）
- 第 1 条主线剧情节点（接入 Flag 系统）
- 存档（localStorage）
- NPC 关系网简版（1-2 个关键 NPC，好感度影响事件选项）
- 命运种子系统（被动事件应对方式埋入因果种子）

### 明确删除（M1 不做）
- 深度分支剧情（M2 才接入）
- 联机 / 多人模式
- 地图可视化
- 人才管理独立界面
- 移动端适配
- NPC 完整关系网（M2 接入，M1 只做1-2个简化版）
- 全部 Buff 类型（M1 只做2个验证）
- 全部大型行动（M1 每种赛道只实现1个）

## Risk Register

| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| 月令制复杂度超预期 | 高 | 高 | 先实现"4时段+8行动"骨架，时段加成和大型行动后续迭代 |
| 被动事件选择式设计耗时 | 中 | 高 | 每个被动事件先写2个选项（积极/保守），冒险选项后续补充 |
| 玩家不理解时段加成 | 中 | 中 | UI中高亮显示当前时段加成，行动效果预览中标注加成量 |
| 大型行动中断体验差 | 低 | 中 | 中断时明确提示惩罚，并给予"继续"或"放弃"的选择 |
| 选项后果设计枯燥（只是数值 +/-）| 中 | 高 | 每个选项后果加 1 句描述性文字增强反馈感 |
| 游戏感薄弱（缺乏目标感） | 中 | 高 | 时间叙事化（年号+季节）+ Buff 系统增强进程感 |
| 资源数值设计失衡 | 中 | 中 | 在15回合内预先推演，时段加成会使数值膨胀，需预留空间 |

## Acceptance（验收条件）

### 用户层
- [ ] 测试者不看说明能在 2 分钟内理解"一个月有4个时段，不同时段做不同事"。
- [ ] 测试者能感知到"在朔日上奏折比在月末效果好"。
- [ ] 测试者能描述出"我的选择产生了后果"（包括被动事件的应对选择）。
- [ ] 测试者有二次游玩意愿（想换一个身份或路线再试）。

### 产品层
- [ ] 所有 2 个身份 × 2 条路线的组合均可完整游玩至终局。
- [ ] 4 个时段均能正常选择行动，时段加成正确结算。
- [ ] 大型行动（占2时段）可正常执行和中断，中断惩罚正确触发。
- [ ] 被动事件触发时弹出选择窗口，所有选项均可正常选择并产生对应后果。
- [ ] 至少 2 个 Buff 可正常触发并产生持续效果。
- [ ] 资源数值在正常游玩中有涨有落，时段加成不会导致数值失控。
- [ ] 胜负触发逻辑正常运行，无死循环。

### 技术层
- [ ] 纯浏览器打开 `index.html` 即可运行，无需服务器。
- [ ] 无 JS 报错（Console 无红色错误）。
- [ ] 最终文件不超过 6 个（index.html / style.css / game.js / actions.json / events.json / README.md）。

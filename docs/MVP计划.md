# MVP 计划 · 权谋录

> 版本目标：在最小实现下验证"身份选择 + 路线发展"的文字 SLG 核心体验是否成立。

---

## 版本目标

- **版本号**：v0.1 — M1 框架原型
- **要验证的核心价值**：
  1. 行动点 + 六赛道行动集的回合结构是否产生足够的策略感和"人生选择"感。
  2. 赛道行动的资源增减是否清晰可读，玩家能看懂自己在"走哪条路"。
  3. 基础框架是否能无缝接入剧情节点（M2 任务）。

## MVP Scope

### 必须交付（M1 框架原型）

| 功能 | 说明 |
|---|---|
| 角色创建界面 | 选择性别 + 选择出身（MVP 实现 3 种：寒门学子 / 边境武将 / 落魄商贾） |
| 赛道选择 | 选择初始人生方向（MVP 实现 2 赛道：官场 / 造反）|
| 资源面板 | 每个赛道显示对应 3-4 种核心资源 + 通用"钱粮" |
| 行动系统 | 每回合 3 个行动点；官场赛道 5 种行动，造反赛道 4 种行动 |
| 行动结果 | 每个行动 → 即时资源增减 + 一句结果描述文字 |
| 被动事件 | 每回合 0-1 个随机事件（10 个事件池，无分支，固定影响） |
| 回合结算 | 资源自然增减（基于已选赛道）→ 胜负判定 |
| 简单结局 | 资源归零 → 覆灭结局；20 回合后 → 简单终局结算文字 |

### 可选交付（时间允许时加入）
- 第 1 条主线剧情节点（接入 Flag 系统）
- 建设系统简版（1 种建设类型）
- 1-2 个可招募人才（简单特长效果）
- 存档（localStorage）

### 明确删除（M1 不做）
- 深度分支剧情（M2 才接入）
- 联机 / 多人模式
- 地图可视化
- 人才管理独立界面
- 移动端适配

## Risk Register

| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| 事件内容写作耗时超预期 | 高 | 高 | MVP 每路线只写 10 节点，用数量控制；先把结构跑通再填内容 |
| 选项后果设计枯燥（只是数值 +/-）| 中 | 高 | 每个选项后果加 1 句描述性文字增强反馈感 |
| 游戏感薄弱（缺乏目标感） | 中 | 高 | 在 UI 上显示"当前回合 / 总回合"和"当前势力状态描述"增强进程感 |
| 资源数值设计失衡 | 中 | 中 | 在10回合设计内用电子表格预先推演一次，保证不过于宽松也不过于残酷 |

## Acceptance（验收条件）

### 用户层
- [ ] 测试者不看说明能在 2 分钟内理解如何开始游戏。
- [ ] 测试者能描述出"我的选择产生了后果"（哪怕是负面后果）。
- [ ] 测试者有二次游玩意愿（想换一个身份或路线再试）。

### 产品层
- [ ] 所有 2 个身份 × 2 条路线的组合均可完整游玩至终局。
- [ ] 5 种资源数值在正常游玩中有涨有落，不会一直单调增长。
- [ ] 胜负触发逻辑正常运行，无死循环。

### 技术层
- [ ] 纯浏览器打开 `index.html` 即可运行，无需服务器。
- [ ] 无 JS 报错（Console 无红色错误）。
- [ ] 最终文件不超过 5 个（index.html / style.css / game.js / events.json / README.md）。
